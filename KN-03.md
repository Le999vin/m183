Beachten Sie die [allgemeinen Informationen zu den Abgaben](./Abgaben.md).

## Lernziele

- Sie können eine laufende PHP-Applikation auf Sicherheitslücken im Sessionhandling analysieren.
- Sie können eine Session-Fixation-Schwachstelle erkennen und mit `session_regenerate_id()` beheben.
- Sie können Passwörter korrekt mit `password_hash()` (Argon2ID) hashen.
- Sie können Session-Cookies mit den Flags `HttpOnly`, `Secure` und `SameSite` absichern.
- Sie können die vier Authentifizierungsfaktor-Kategorien (Wissen, Besitz, Inhärenz, Ort) erklären und von MFA abgrenzen.

## C) Session-Fixation-Angriff demonstrieren

### Hintergrund

Bei **Session Fixation** nutzt ein Angreifer aus, dass die Session-ID nach dem Login nicht erneuert wird. Er kann eine Session-ID vorab festlegen, das Opfer damit einloggen lassen und dann diese bekannte ID verwenden, um auf die Session des Opfers zuzugreifen.

### Aufgabe

1. Öffnen Sie die App in **zwei verschiedenen Browsern** (z.B. Chrome und Firefox, oder Chrome und Chrome Inkognito).
2. Lesen Sie im ersten Browser die Session-ID aus: **DevTools** (F12) → **Application** → **Cookies** → `PHPSESSID`.
3. Kopieren Sie diese Session-ID.
4. Setzen Sie im zweiten Browser denselben Cookie-Wert (DevTools → Application → Cookies → Doppelklick auf den PHPSESSID-Wert → Wert ersetzen).
5. Loggen Sie sich im **ersten Browser** ein (beliebiger Benutzername und Passwort).
6. Laden Sie den **zweiten Browser** neu – was sehen Sie?

Beantworten Sie folgende Fragen schriftlich:

1. Was ist passiert? Konnte der zweite Browser auf die Session des ersten zugreifen?
2. Warum ist das ein Sicherheitsproblem?
3. Welche eine Massnahme hätte diesen Angriff verhindert?

**Abgabe:**

- Screenshot des zweiten Browsers nach Schritt 6 (Session-Daten sichtbar).
- Schriftliche Antworten auf die drei Fragen.

## D) Sicherheitslücken beheben

Erstellen Sie eine Kopie der `index.php`, um die Originaldatei zu erhalten:

```bash
cp ~/m183/"2 Sessionhandling, Authentifizierung und Autorisierung"/Sessionhandling/AufgabeSource/index.php \
   ~/m183/"2 Sessionhandling, Authentifizierung und Autorisierung"/Sessionhandling/AufgabeSource/index_original.php
```

Öffnen Sie die Datei zum Bearbeiten:

```bash
nano ~/m183/"2 Sessionhandling, Authentifizierung und Autorisierung"/Sessionhandling/AufgabeSource/index.php
```

Implementieren Sie folgende Fixes:

### Fix 1 – Session-ID nach Login erneuern

Fügen Sie direkt nach dem erfolgreichen Login-Block folgende Zeile ein:

```php
session_regenerate_id(true);
```

Dieser Befehl erzeugt nach dem Login eine neue Session-ID und macht damit Session-Fixation unmöglich. Der Parameter `true` löscht die alte Session-Datei auf dem Server.

### Fix 2 – Passwort-Hashing mit Argon2ID

Die aktuelle App prüft das Passwort gar nicht. In einer echten Applikation würden Passwörter gehasht gespeichert. Ergänzen Sie die Login-Logik so, dass das Passwort geprüft wird. Fügen Sie oberhalb des `session_start()` ein Array mit gehashten Passwörtern ein:

```php
// Gehashte Passwörter (in einer echten App in der Datenbank)
$users = [
    'admin'  => password_hash('geheim123', PASSWORD_ARGON2ID),
    'alice'  => password_hash('passwort', PASSWORD_ARGON2ID),
];
```

Ersetzen Sie den Login-Block durch:

```php
if(isset($_POST['login'])) {
    $username = $_POST['username'];
    $password = $_POST['password'];

    if(isset($users[$username]) && password_verify($password, $users[$username])) {
        session_regenerate_id(true);
        $_SESSION['username'] = $username;
        $_SESSION['role'] = ($username === 'admin') ? 'admin' : 'user';
    }
}
```

### Fix 3 – Cookie-Flags setzen

Fügen Sie direkt nach `session_start()` folgende Zeilen ein:

```php
session_set_cookie_params([
    'lifetime' => 900,        // 15 Minuten Timeout
    'secure'   => true,       // Nur über HTTPS senden
    'httponly' => true,       // Kein Zugriff via JavaScript
    'samesite' => 'Strict',   // Kein Cross-Site-Senden
]);
```

Speichern Sie die Datei (Nano: `Ctrl+O`, `Enter`, `Ctrl+X`). Der Container liest die Datei direkt aus dem gemounteten Volume – ein Neustart ist nicht nötig.

Laden Sie die App im Browser neu und prüfen Sie die Cookie-Flags in den DevTools:

```
DevTools → Application → Cookies → PHPSESSID
```

Die Spalten **HttpOnly** und **Secure** müssen angehakt sein.

Beantworten Sie folgende Fragen schriftlich:

1. Was bewirkt `HttpOnly` auf einem Cookie? Gegen welchen Angriff schützt dieses Flag?
2. Was bewirkt `SameSite=Strict`? Gegen welchen Angriff schützt dieses Flag?
3. Warum wurde `password_hash()` mit `PASSWORD_ARGON2ID` und nicht mit MD5 oder SHA-1 verwendet?
