## A) WebGoat starten (Pflicht, zuerst lösen) (10%)

### Schritt 1 – Port freigeben

WebGoat läuft auf Port **8080**. Öffnen Sie diesen Port in der bestehenden Sicherheitsgruppe `m183-sg`.

Navigieren Sie zu **EC2** → **Security Groups** → `m183-sg` → **Inbound rules** → **Edit inbound rules**. Fügen Sie folgende Regel hinzu:

| Type | Protocol | Port range | Source |
|------|----------|------------|--------|
| Custom TCP | TCP | 8080 | **My IP** |

Klicken Sie auf **Save rules**.

> ⚠️ App-Ports nur für **My IP** öffnen, nicht für `0.0.0.0/0`.

### Schritt 2 – WebGoat starten

Verbinden Sie sich via SSH mit Ihrer EC2-Instanz und starten Sie WebGoat:

```bash
docker run -d --name webgoat -p 8080:8080 webgoat/webgoat
```

Prüfen Sie, ob der Container läuft:

```bash
docker ps
```

Öffnen Sie im Browser: `http://<IHRE-EC2-IP>:8080/WebGoat`

Klicken Sie auf **Register new user** und erstellen Sie ein Konto. Notieren Sie Benutzername und Passwort – Sie brauchen die Zugangsdaten in einigen Aufgaben.

> Falls die Seite nicht erreichbar ist: Security Group prüfen (Port 8080, My IP). Falls sich Ihre IP geändert hat, passen Sie die Regel an.

## B) SQL Injection (18%)

> ⚠️ BEI ALLEN WEBGOAT ABGABEN MUESSEN SIE WEBGOAT **LIVE** MIT **IHRER** URL ZEIGEN!
>

### Hintergrund

SQL Injection (SQLi) gehört zu den ältesten und gefährlichsten Webangriffstechniken. Sie entsteht, wenn eine Applikation Benutzereingaben ungefiltert direkt in SQL-Abfragen einfügt. Der Angreifer kann dadurch die Abfrage strukturell verändern – mit dem Ziel, Daten zu lesen, zu manipulieren oder die Authentifizierung zu umgehen.

**Wie entsteht die Schwachstelle?**

Unsicherer Code (Java-Beispiel):
```java
String query = "SELECT * FROM users WHERE name = '" + userInput + "'";
```

Gibt der Benutzer `Smith' OR '1'='1` ein, entsteht folgendes SQL:
```sql
SELECT * FROM users WHERE name = 'Smith' OR '1'='1'
```

Da `'1'='1'` immer wahr ist, gibt die Datenbank alle Zeilen zurück – egal welcher Name in der Tabelle steht. Die ursprüngliche Bedingung wird dadurch bedeutungslos.

---

### B1 – Login Bypass

Navigieren Sie in WebGoat zu **A3 Injection → SQL Injection (intro)**.

Arbeiten Sie sich durch die einleitenden Informationsseiten (kein Input nötig, weiterklicken) bis Sie zur Login-Bypass-Aufgabe kommen. Das Formular hat zwei Felder (z.B. Name und Passwort oder Name und TAN). Ziel: Einloggen, ohne das korrekte Passwort zu kennen.

**Vorgehen:**

1. Geben Sie im ersten Feld einen Namen ein (z.B. `Smith`).
2. Geben Sie im zweiten Feld folgenden Payload ein:

```
' OR '1'='1
```

3. Klicken Sie auf **Submit**.

Falls dieser Payload nicht funktioniert, probieren Sie Varianten:

```
' OR 1=1 --
```
```
anything' OR 'x'='x
```

WebGoat zeigt bei Erfolg eine grüne Bestätigungsmeldung.

**Was passiert im Hintergrund?**

Das ursprüngliche SQL-Statement:
```sql
SELECT * FROM users WHERE name = 'Smith' AND password = 'eingabe'
```

Wird durch den Payload zu:
```sql
SELECT * FROM users WHERE name = 'Smith' AND password = '' OR '1'='1'
```

Da `OR '1'='1'` immer wahr ist, liefert die Abfrage alle Benutzer zurück – und die Applikation interpretiert das als erfolgreichen Login.

### B2 – Query Chaining: Integrität kompromittieren

Navigieren Sie in WebGoat zu **A3 Injection → SQL Injection (intro) → Compromising Integrity with Query chaining**.

Bei **Query Chaining** werden mehrere SQL-Anweisungen hintereinander in eine einzige Eingabe verpackt. Das Semikolon `;` markiert das Ende einer SQL-Anweisung – direkt dahinter kann eine zweite Anweisung beginnen, ohne dass eine neue Zeile nötig ist:

```sql
erster Befehl; zweiter Befehl; dritter Befehl
```

Ein Angreifer kann so nicht nur die ursprüngliche Abfrage manipulieren, sondern zusätzliche Befehle einschleusen – zum Beispiel ein `UPDATE`, das Daten in der Datenbank verändert, oder ein `INSERT`, das einen neuen Benutzer anlegt.

Beispiel: Wenn die originale Abfrage eine Mitarbeiterliste lädt, kann mit Query Chaining gleichzeitig das Gehalt eines anderen Mitarbeiters geändert werden:
```sql
'; UPDATE employees SET salary = 9999999 WHERE name = 'admin'; --
```

Folgen Sie den Anweisungen in WebGoat, lösen Sie die Aufgabe und notieren Sie:
- Den verwendeten Payload (mit `;`).
- Welche Veränderung Sie in der Datenbank damit bewirkt haben.

## C) Cross-Site Scripting (XSS) (18%)

### Hintergrund

Cross-Site Scripting (XSS) ist eine Schwachstelle, bei der ein Angreifer schadhaften JavaScript-Code in eine Webseite einschleust, der dann im Browser anderer Benutzer ausgeführt wird. Die Applikation gibt die Eingabe des Angreifers unkodiert aus – der Browser des Opfers interpretiert sie als legitimen Code.

**Drei XSS-Typen im Überblick:**

| Typ | Mechanismus | Wen trifft es? |
|-----|-------------|----------------|
| **Reflected XSS** | Payload in URL; Server spiegelt ihn sofort zurück | Nur wer den manipulierten Link öffnet |
| **Stored XSS** | Payload in Datenbank gespeichert (z.B. Kommentar) | Alle Benutzer, die die Seite laden |
| **DOM-based XSS** | Payload wird clientseitig durch JavaScript in den DOM geschrieben – der Server sieht ihn nie | Nur wer den manipulierten Link öffnet |

**Warum ist `<script>` oft blockiert?**

Viele Browser und Sicherheitsfilter blockieren `<script>`-Tags in Formularfeldern. Angreifer weichen deshalb auf HTML-Event-Handler aus:

```html
<img src="x" onerror="alert('XSS')">
```

Der `onerror`-Handler wird ausgeführt, sobald das Bild nicht geladen werden kann – bei `src="x"` ist das garantiert der Fall.

---

### C1 – Reflected XSS & DOM-based XSS

#### C1a – Try It! Reflected XSS

Suchen Sie in der Lektion die Aufgabe **«Try It! Reflected XSS»**. WebGoat stellt ein Eingabefeld bereit (z.B. für eine Kreditkartennummer), dessen Wert ungefiltert in die Seitenantwort zurückgespiegelt wird.

**Vorgehen:**

1. Geben Sie zunächst einen normalen Wert ein (z.B. `1234`) und beobachten Sie, wo er in der Seite erscheint.
2. Öffnen Sie **DevTools → Network**, lösen Sie die Anfrage erneut aus und prüfen Sie im **Response**-Tab: Steht Ihre Eingabe direkt im HTML-Quelltext?
3. Injizieren Sie einen JavaScript-Payload in das Eingabefeld:

```html

```

4. Wird ein Alert ausgelöst? → Reflected XSS bestätigt. Notieren Sie den Payload.

#### C1b – Identify potential for DOM-Based XSS

Suchen Sie in der Lektion die Aufgabe **«Identify potential for DOM-Based XSS»**.

Bei **DOM-based XSS** passiert die Injection vollständig auf der Client-Seite: JavaScript liest einen Wert (z.B. aus der URL) und schreibt ihn ohne Filterung in den DOM. Der Server-Response enthält den Payload gar nicht – er wird erst im Browser «eingebaut». Das macht ihn für serverseitige Filter unsichtbar.

Typisch verwundbare JavaScript-Muster:
```javascript
document.getElementById('output').innerHTML = location.hash.slice(1); // URL-Fragment direkt in den DOM
document.write(decodeURIComponent(location.search));                   // Query-Parameter direkt ausgegeben
```

WebGoat zeigt Ihnen Code-Ausschnitte und fragt, welche Stellen potenziell für DOM-based XSS anfällig sind. Analysieren Sie die Ausschnitte und markieren Sie die verwundbaren Punkte gemäss den WebGoat-Anweisungen.

Notieren Sie: Welches JavaScript-Konstrukt war verwundbar und warum?

## D) CSRF – Cross-Site Request Forgery (18%)

### Hintergrund

Bei **CSRF** (Cross-Site Request Forgery) bringt ein Angreifer den Browser eines eingeloggten Opfers dazu, im Hintergrund eine Aktion auf einer fremden Webseite auszuführen – ohne dass das Opfer es bemerkt.

**Wie funktioniert das?**

Das Opfer ist auf `bank.ch` eingeloggt und hat einen Session-Cookie. Es öffnet nebenbei eine Angreifer-Seite. Diese Seite enthält ein verstecktes Formular, das automatisch an `bank.ch/transfer` sendet:

```html
<!-- Angreifer-Seite: gewinne.html -->
<form id="f" action="http://bank.ch/transfer" method="POST">
  <input type="hidden" name="to" value="angreifer-konto">
  <input type="hidden" name="amount" value="9999">
</form>
<script>document.getElementById('f').submit();</script>
```

Der Browser schickt den `bank.ch`-Session-Cookie automatisch mit – er gehört zur Domain, nicht zur Seite, die die Anfrage auslöst. `bank.ch` sieht eine Anfrage mit gültigem Cookie und führt die Überweisung aus.

**Warum funktioniert das?** Browser schicken Cookies automatisch zu ihrer Domain, egal von welcher Seite die Anfrage stammt. Das ist das Grundprinzip von Session-Cookies. CSRF missbraucht genau dieses Verhalten.

**Gegenmassnahme – CSRF-Token:** Ein zufälliger, geheimer Wert wird bei jeder sensitiven Anfrage mitgeschickt und serverseitig geprüft. Den Token kennt nur die legitime Seite (eingebettet im HTML) – eine Angreifer-Seite kann ihn wegen der Same-Origin-Policy nicht auslesen.

## E) Broken Access Control – IDOR (18%)

### Hintergrund

**Broken Access Control** bedeutet, dass eine Applikation nicht ausreichend prüft, ob ein Benutzer auf eine bestimmte Ressource zugreifen darf. **IDOR** (Insecure Direct Object Reference) ist die häufigste Form: Datenbankobjekte wie Profile, Dokumente oder Bestellungen werden über vorhersehbare IDs angesprochen – und die Applikation prüft nicht, ob der Anfragende auch der Eigentümer ist.

**Beispiel:**

```
GET /profile?id=1234   ← eigenes Profil
GET /profile?id=1235   ← fremdes Profil — einfach ID + 1!
```

Wenn der Server bei `id=1235` nicht prüft, ob der eingeloggte Benutzer Eigentümer dieses Profils ist, liegt IDOR vor. Der Angreifer braucht keine erhöhten Rechte – er ist eingeloggt, aber sieht Daten, die ihm nicht gehören.

IDOR ist laut OWASP 2021 unter **A01 – Broken Access Control** eingestuft – erstmals auf Platz 1, weil es in 94% aller getesteten Applikationen vorkommt.

## F) Broken Authentication – JWT Tokens (18%)

### Hintergrund

**JWT** (JSON Web Token) ist ein verbreiteter Standard für zustandslose Authentifizierungstoken. Ein JWT besteht aus drei Base64url-codierten Teilen, getrennt durch Punkte:

```
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJ1c2VyIjoiam9obiJ9.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c
|____________Header___________________|.__Payload___.|._________Signatur___________________________|
```

- **Header**: Signaturalgorithmus (`alg`) und Typ (`typ`). Beispiel: `{"alg":"HS256","typ":"JWT"}`
- **Payload**: Die eigentlichen Nutzdaten (z.B. Benutzername, Rolle, Ablaufzeit `exp`). Beispiel: `{"user":"john","role":"user"}`
- **Signatur**: Kryptografischer Beweis, dass Header und Payload nicht verändert wurden.

Der Server prüft beim Eingang eines Tokens die Signatur. Stimmt sie nicht, wird der Token abgelehnt. Das verhindert, dass ein Angreifer den Payload (z.B. die Rolle) eigenmächtig verändert.

**Die `alg:none`-Schwachstelle:**

Manche fehlerhafte Implementierungen akzeptieren `"alg":"none"` im Header – was bedeutet: «Keine Signatur nötig.» Ein Angreifer kann dann:

1. Den Payload dekodieren (Base64url ist kein Verschlüsselungsverfahren – jeder kann ihn lesen).
2. Den Payload verändern (z.B. `"role":"admin"` setzen).
3. Den Header auf `"alg":"none"` ändern.
4. Einen Token ohne Signatur einreichen: `<header>.<payload>.`
5. Der Server akzeptiert ihn, weil er glaubt, keine Signaturprüfung sei konfiguriert.

---

### Aufgabe

**Schritt 1 – JWT-Struktur analysieren**

Führen Sie den Login-Schritt in der WebGoat-Lektion aus. Ein Token wird ausgegeben. Kopieren Sie ihn und analysieren Sie ihn auf [https://jwt.io](https://jwt.io):

- Fügen Sie den Token in das Feld «Encoded» ein.
- Lesen Sie den Payload: Welchen Benutzernamen enthält er? Welche Rolle oder Berechtigung? Gibt es einen Ablaufzeitpunkt (`exp`)?
- Notieren Sie die Beobachtungen.

**Schritt 2 – alg:none-Angriff**



1. Dekodieren Sie den Header. Öffnen Sie die **Browser-Konsole** (F12) und führen Sie aus:
   ```javascript
   atob("eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9")
   // Ergibt: {"alg":"HS256","typ":"JWT"}
   ```

2. Erstellen Sie einen neuen Header mit `alg:none`:
   ```json
   {"alg":"none","typ":"JWT"}
   ```

3. Verändern Sie den Payload: Setzen Sie die Rolle auf `admin` oder den Benutzernamen auf einen Administratornamen (WebGoat gibt Hinweise).

4. Base64url-enkodieren Sie Header und Payload ohne Padding (kein `=` am Ende). In der Browser-Konsole:
   ```javascript
   btoa('{"alg":"none","typ":"JWT"}').replace(/=/g, '').replace(/\+/g, '-').replace(/\//g, '_')
   btoa('{"user":"admin","role":"admin"}').replace(/=/g, '').replace(/\+/g, '-').replace(/\//g, '_')
   ```

5. Setzen Sie den manipulierten Token zusammen – mit **leerem Signatur-Teil**, aber dem abschliessenden Punkt:
   ```
   <neuer-header>.<neuer-payload>.
   ```

6. Reichen Sie den Token in WebGoat ein. Bei Erfolg erscheint eine grüne Bestätigung.

Folgen Sie den WebGoat-Schritten und notieren Sie den vollständigen manipulierten Token.
